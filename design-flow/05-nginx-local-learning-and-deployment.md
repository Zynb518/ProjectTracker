# 当前项目 Nginx 本地学习与部署设计

## 1. 文档说明

本文档用于记录当前项目的 `Nginx` 本地学习与落地方案，目标是在开发机上通过真实项目建立一套“统一入口”的最小闭环。

本次方案不追求一次性覆盖全部 `Nginx` 能力，而是围绕当前项目最需要的访问链路展开：

- 局域网设备通过固定 IP 访问项目
- `Nginx` 负责作为浏览器访问入口
- `Nginx` 负责把前端页面请求转发到本机 `Vite` 开发服务器
- `Nginx` 负责把 `/api` 请求转发到本机 Drogon 后端

本文档作为后续实操步骤和学习记录的基线文档，后续每个阶段的操作结果、问题和调整结论都继续沉淀在本文件中。

## 2. 当前已确认的目标

### 2.1 访问目标

- 当前阶段在本机练习，不先进入虚拟机部署
- 访问入口固定为 `http://192.168.40.74`
- 允许同一局域网内其他设备访问该地址
- 当前阶段只做 `HTTP`，暂不引入 `HTTPS`

### 2.2 项目接入目标

- 使用当前项目的真实前端，而不是临时演示页面
- 使用当前项目的真实后端 Drogon 服务，而不是假接口
- 当前阶段前端不强制打包，继续使用 `Vite` 开发服务器承接页面请求
- 假定后端已经可以独立运行，并能通过 `127.0.0.1:8080` 提供接口

### 2.3 学习目标

当前学习不是把 `Nginx` 当成纯理论知识点拆开理解，而是让学习过程和真实项目接入同步推进。

完成本阶段后，应达到以下结果：

- 能说明浏览器请求为什么会先到 `Nginx`
- 能说明 `Nginx` 为什么既能转发前端页面，也能转发 `/api`
- 能通过局域网 IP 打开当前项目页面
- 能让前端真实接口请求通过 `Nginx` 到达后端
- 能在保留前端热更新的情况下继续使用统一入口访问项目

## 3. 目标架构

当前阶段采用单机本地开发部署形态，结构如下：

```text
局域网设备浏览器
        |
        v
http://192.168.40.74
        |
        v
      Nginx:80
      /      \
     /        \
前端页面        /api
  |             |
  v             v
Vite dev      Drogon
server        127.0.0.1:8080
```

该结构与当前项目现状一致：

- 前端请求以根路径为基准
- 开发阶段已经约定 `/api` 代理到 `127.0.0.1:8080`
- 后端当前配置监听 `0.0.0.0:8080`
- 当前前端仍在持续修改，需要保留热更新体验

因此本次学习并不是额外造一套全新的部署模式，而是先把开发期的访问关系收敛为统一入口；等前端相对稳定后，再把前端目标从 `Vite dev server` 切换为打包后的静态目录。

## 4. 当前项目与 Nginx 的对应关系

### 4.1 前端

- 前端构建工具为 `Vite`
- 当前阶段为了保留热更新，前端页面先通过 `Vite dev server` 提供
- 前端稳定后，再切换为 `Nginx` 直接托管打包后的静态产物
- 前端请求默认以 `/` 为根路径
- 前端请求携带 Cookie，用于保持登录态

### 4.2 后端

- 后端为 `Drogon`
- 当前默认监听配置为 `0.0.0.0:8080`
- 登录态采用 `Session/Cookie`
- 生产式访问更适合通过 `Nginx` 统一入口对外暴露

### 4.3 Nginx 在当前项目中的作用

在当前项目中，`Nginx` 的核心职责限定为以下几项：

- 监听 `80` 端口，对外提供统一入口
- 把前端页面请求转发到本机 `Vite` 开发服务器
- 把 `/api` 请求反向代理到本机后端
- 在当前阶段作为后续“静态托管版”部署的过渡原型

当前阶段明确不纳入范围的内容：

- `HTTPS`
- 负载均衡实战
- 多机部署
- 复杂缓存策略
- `systemd` 托管与开机自启

## 7. 初步风险与注意事项

### 7.1 局域网访问风险

- 如果系统防火墙未放行 `80` 端口，其他设备将无法访问
- 如果本机 IP 发生变化，访问地址需要同步调整

### 7.2 本机服务冲突风险

- 若已有其他服务占用 `80` 端口，`Nginx` 将无法正常绑定
- 若 `Vite dev server` 未正确监听可访问地址，前端页面代理会失败
- 若后端未稳定运行在 `127.0.0.1:8080`，`/api` 代理阶段会失败

### 7.3 单页应用路由风险

- 当前阶段若 `Vite dev server` 主机校验或监听地址限制过严，局域网访问可能失败
- 后续切换为静态托管时，若未补充 `try_files` 路由回退，刷新详情页或列表页深链接时会出现 `404`

## 9. 实操记录

### 9.1 第一步：让 Vite dev server 可作为 Nginx 上游

目标：

- 先让前端开发服务器具备被 `Nginx` 稳定代理的前提
- 保留热更新，不要求当前阶段先打包前端

本次调整：

- 在 `frontend/vite.config.ts` 中把开发服务器监听地址改为 `0.0.0.0`
- 固定开发端口为 `5173`
- 开启 `strictPort`
- 把 `allowedHosts` 从仅允许 `www.project.test` 改为当前学习阶段可接受局域网 IP 访问

为什么要这样做：

- `host: '0.0.0.0'` 让 `Vite dev server` 不只监听本机回环地址，便于 `Nginx` 和局域网设备访问
- 固定 `5173` 端口后，后续 `Nginx` 的上游目标可保持稳定
- `strictPort: true` 避免 Vite 自动漂移到其他端口，导致 `Nginx` 配置失效
- 当前原始配置只允许 `www.project.test`，不适合 `192.168.40.74` 这类 IP 访问

下一步验证：

- 启动前端开发服务器
- 确认本机可通过 `127.0.0.1:5173` 访问
- 确认本机或局域网设备可通过 `192.168.40.74:5173` 访问

结果：

- 前端开发服务器重启后，可通过 `http://192.168.40.74:5173/` 正常访问

### 9.2 第二步：接入 Nginx 统一入口

目标：

- 先不拆分 `/api`
- 先让 `Nginx` 监听 `80` 端口，并把页面请求统一转发到 `Vite dev server`

本次配置：

- 新增站点配置文件 `project-tracker.local`
- 使用 `server { listen 80; ... }` 建立第一版站点
- 使用 `location / { proxy_pass http://127.0.0.1:5173; ... }` 把页面请求转发给前端开发服务器
- 保留 `Upgrade` 和 `Connection` 头，保证开发期热更新链路可用

遇到的问题：

- 首次创建 `sites-enabled` 软链接时，链接目标错误，形成了自引用循环
- 错误表现为 `Too many levels of symbolic links`
- 修正方式是删除错误链接，并重新创建指向 `/etc/nginx/sites-available/project-tracker.local` 的绝对路径软链接

验证结果：

- `sudo nginx -t` 已通过
- 通过 `sudo nginx -s reload` 完成配置重载
- 访问 `http://192.168.40.74` 已可打开当前前端页面

当前结论：

- `Nginx:80 -> Vite:5173` 这段链路已打通
- 下一步开始把 `/api` 请求从当前默认链路中拆出来，交给 `Nginx` 直接反向代理到 `Drogon`
