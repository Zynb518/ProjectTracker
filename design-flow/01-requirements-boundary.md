# 项目进度跟踪软件需求边界文档（重写版）

## 1. 项目基本信息

- 项目名称：`项目进度跟踪软件`
- 项目定位：`企业内部使用的轻量项目进度跟踪工具`
- 使用模式：`支持单人自用项目，也支持团队协作项目`
- 开发方式：`个人独立开发`
- 目标技术栈：`C++ Drogon + PostgreSQL + HTML/CSS/JavaScript + Nginx`

## 2. 产品目标

本系统用于帮助企业内部成员统一管理项目执行过程，减少 Excel、聊天记录和口头同步带来的信息分散问题。

系统重点解决以下问题：

- 项目从创建到完成的过程缺乏统一记录
- 阶段规划、任务分配和进度更新分散在多个地方
- 项目负责人难以及时看到阶段推进情况和成员任务情况
- 普通成员缺少清晰的个人任务入口和进度提交通道

第一版的目标不是构建完整项目管理平台，而是先完成“项目规划 + 阶段推进 + 子任务跟踪 + 甘特图展示”这条核心闭环。

## 3. 适用范围

- 系统面向企业内部使用，不对外开放注册
- 第一版仅支持 PC 端网页访问
- 支持单人项目和团队项目两种使用方式
- 当前重点描述团队使用场景
- 不考虑多组织、多租户场景
- 不考虑高并发场景，满足企业内部日常协作即可

## 4. 目标用户

### 4.1 管理员

职责：

- 维护用户信息
- 查看和管理全部项目
- 处理系统级管理事务
- 在需要时参与项目并担任项目负责人

### 4.2 项目经理

职责：

- 创建项目
- 在自己负责的项目中管理项目成员
- 规划项目阶段节点
- 创建和分配子任务
- 查看项目整体进度、阶段进度和成员任务情况

说明：

- 项目经理是系统级业务角色，不等同于系统管理员
- 项目经理创建项目后，会自动成为该项目负责人

### 4.3 普通员工

职责：

- 参与项目并查看自己相关的项目内容
- 查看自己负责的子任务
- 提交子任务进度更新和备注
- 在个人自用场景下创建和管理自己的项目

## 5. 权限边界

### 5.1 系统级权限

- 管理员拥有全局查看和管理权限
- 对管理员或项目经理创建的团队项目，管理员可以直接执行项目内操作，但仍需遵守统一业务规则限制
- 对普通员工创建的个人自用项目，管理员仅保留查看和删除权限，不允许进行项目内部修改
- 项目经理可以创建和管理自己负责的团队项目
- 普通员工只能访问自己可见范围内的项目与任务

### 5.2 项目内权限

- 任何用户都可以创建项目
- 项目创建者自动成为该项目的项目负责人
- 项目负责人负责本项目的阶段规划和子任务分配
- 管理员可以直接对团队项目添加或移除成员，但仍需遵守统一业务规则限制
- 非管理员场景下，只有具备管理员或项目经理身份的项目负责人，才能添加或移除项目成员
- 普通员工不能添加项目成员
- 项目成员可以查看自己参与的项目，并处理自己负责的子任务

### 5.3 第一版的成员管理限制

- 管理员和项目经理创建的项目可以发展为团队项目
- 普通员工创建的项目在第一版固定视为个人自用项目，任何角色都不允许将其转成团队项目或新增其他成员
- 对普通员工创建的个人自用项目，管理员不允许执行项目内部修改动作，例如添加成员、调整负责人、修改阶段节点或修改子任务
- 第一版成员加入方式采用“项目负责人直接添加”，不实现“邀请后等待对方确认”的流程
- 项目成员名下若仍存在未完成子任务，则该成员不能被直接移出项目

## 6. 核心信息结构

系统采用 `项目 -> 阶段节点 -> 子任务` 的层次模型。

### 6.1 项目

项目表示一个需要被持续跟踪的整体目标。

项目可以是：

- 单人项目
- 团队协作项目
- 第一版创建项目时必须填写 `planned_start_date` 和 `planned_end_date`

### 6.2 阶段节点

阶段节点表示项目执行路径中的一个阶段性目标，具有顺序性。

阶段节点的特点：

- 隶属于某一个项目
- 在项目内按线性顺序组织
- 前端支持拖拽调整阶段顺序
- 可以只作为阶段 / 里程碑存在，不要求必须包含子任务
- 必须填写 `planned_start_date` 和 `planned_end_date`，用于甘特图展示
- 阶段节点的时间区间必须落在所属项目的时间区间内

### 6.3 子任务

子任务表示阶段节点下的具体执行工作项。

子任务的特点：

- 必须归属于某一个阶段节点
- 同一阶段下可以有多个子任务
- 同一阶段下的多个子任务一般视为可并行执行
- 每个子任务只能指定一个负责人
- 如果创建时暂未指定负责人，则默认指派给项目负责人
- 第一版创建子任务时必须填写 `planned_start_date` 和 `planned_end_date`
- 子任务的时间区间必须落在所属阶段节点的时间区间内

### 6.4 子任务进度记录

子任务进度记录用于保存子任务推进过程中的历史更新。

每条进度记录可以包含：

- 当前进度百分比
- 当前状态
- 备注或进度说明
- 更新时间和操作人

## 7. 典型使用流程

以下流程默认按常规项目执行场景描述，主要以项目负责人和子任务负责人为主要操作者；管理员仅在允许介入的团队项目中，可在遵守统一业务规则限制的前提下直接介入。

### 7.1 项目创建

1. 用户创建项目并填写项目基本信息、`planned_start_date` 和 `planned_end_date`
2. 系统自动将创建者设为项目负责人，并同步写入项目成员关系
3. 用户可选择空项目开始，也可选择系统预置模板快速生成多个阶段节点

### 7.2 团队准备

1. 具备成员管理权限的操作者根据项目类型决定是否添加成员；普通员工创建的个人自用项目不允许添加其他成员
2. 团队项目准备完成后，进入阶段规划和任务拆解

### 7.3 阶段规划

1. 项目负责人创建阶段节点并填写 `planned_start_date` 和 `planned_end_date`
2. 阶段节点的时间区间必须被项目时间区间包含
3. 项目负责人可通过拖拽调整阶段顺序
4. 用户点击阶段节点后，可查看节点详情卡片

### 7.4 任务拆解与分配

1. 项目负责人在某个阶段节点下创建多个子任务，并填写子任务 `planned_start_date` 和 `planned_end_date`
2. 子任务的时间区间必须被所属阶段节点时间区间包含
3. 创建子任务时必须指定负责人；若未明确指定，则默认由项目负责人负责
4. 项目负责人后续可以重新分派子任务负责人

### 7.5 任务推进

1. 子任务负责人在“我的子任务”页面查看自己负责的任务
2. 子任务负责人进入任务进度页面提交更新
3. 子任务进度支持多次更新，并保留历史记录
4. 第一版进度百分比以 `10%` 为最小粒度，即仅允许 `0、10、20 ... 100`
5. 子任务状态采用 `未开始 / 进行中 / 已完成 / 已延期`
6. 子任务的开始信号、延期判定和撤销完成后的状态回退统一按第 8 节规则执行

### 7.6 阶段完成

1. 阶段节点的 `已完成` 采用“项目负责人手动确认”的方式
2. 当阶段节点下没有子任务时，项目负责人可以手动开始该阶段，也可以直接将该阶段标记为完成
3. 当阶段节点下存在子任务时，只有当该阶段当前为 `未开始`，且任意一个子任务进入 `进行中 / 已延期 / 已完成`，该阶段才自动进入 `进行中`
4. 当阶段节点下存在子任务时，只有在全部子任务都已完成后，项目负责人才能将该阶段标记为完成
5. 已完成的阶段节点默认完全冻结，不允许直接修改任何字段，也不允许增删改其下子任务；如需继续修改，需先执行“撤销完成”
6. 阶段节点顺序表示计划推进顺序，不表示系统级强依赖关系
7. 下级子任务的自动联动只允许将当前 `未开始` 的阶段推进为 `进行中`，不会自动回写其他状态
8. 阶段节点的状态判定、撤销完成后的回退和是否联动后续阶段，统一按第 8 节规则执行

### 7.7 项目完成

1. 项目的 `已完成` 采用“项目负责人手动确认”的方式
2. 当项目下没有任何阶段节点时，项目负责人可以手动开始项目，也可以直接手动完成项目
3. 当项目下存在阶段节点时，只有当项目当前为 `未开始`，且任意一个阶段节点进入 `进行中 / 已延期 / 已完成`，项目才自动进入 `进行中`
4. 当项目下存在阶段节点时，只有在全部阶段节点都已完成后，项目负责人才能将项目标记为完成
5. 已完成的项目默认完全冻结，不允许直接修改任何字段，也不允许继续调整其中阶段节点或子任务；如需继续修改，需先执行项目“撤销完成”
6. 下级阶段节点的自动联动只允许将当前 `未开始` 的项目推进为 `进行中`，不会自动回写其他状态
7. 项目的状态判定和撤销完成后的回退统一按第 8 节规则执行

## 8. 核心业务规则

- 一个项目可以包含零个或多个阶段节点
- 一个阶段节点只能属于一个项目
- 一个阶段节点可以包含零个或多个子任务
- 一个子任务只能属于一个阶段节点
- 一个项目可以包含多个项目成员
- 一个项目只能有一个项目负责人
- 项目创建者自动成为该项目负责人，并必须同步作为项目成员存在
- 系统角色分为：管理员、项目经理、普通员工
- 项目状态采用：`未开始 / 进行中 / 已完成 / 已延期`
- 阶段节点状态采用：`未开始 / 进行中 / 已完成 / 已延期`
- 子任务状态采用：`未开始 / 进行中 / 已完成 / 已延期`
- 项目必须填写 `planned_start_date` 和 `planned_end_date`
- 阶段节点必须填写 `planned_start_date` 和 `planned_end_date`
- 子任务必须填写 `planned_start_date` 和 `planned_end_date`
- 阶段节点的时间区间必须被所属项目时间区间包含，包含边界值
- 子任务的时间区间必须被所属阶段节点时间区间包含，包含边界值
- 子任务负责人必须是该项目内的成员
- 一个子任务只能有一个负责人
- 子任务创建时若未指定负责人，则默认由项目负责人担任负责人
- 子任务允许多次推进，并保留推进历史
- 子任务进度百分比在第一版按 `10%` 粒度录入
- 已完成的项目、阶段节点、子任务默认完全冻结，不允许直接修改任何字段
- 如需继续修改已完成对象，需先执行“撤销完成”
- “撤销完成”和“编辑”是两个分离动作，必须先撤销完成，再进行任何编辑
- 已完成的项目和阶段节点不允许继续新增、删除或调整下级对象；已完成的子任务不允许继续提交进度
- 第一版删除采用硬删除；删除项目时级联删除其成员、阶段节点、子任务和进度记录；删除阶段节点时级联删除其下子任务和进度记录；删除子任务时级联删除其进度记录
- 项目负责人可以查看和管理本项目全部阶段节点与子任务
- 管理员可以直接向团队项目中添加或移除成员，但仍需遵守统一业务规则限制
- 非管理员场景下，只有具备管理员或项目经理身份的项目负责人，才能向项目中添加成员
- 非管理员场景下，只有具备管理员或项目经理身份的项目负责人，才能将成员移出项目
- 项目成员若仍有未完成子任务，则不能被移出项目
- 管理员可以在必要时直接调整团队项目的负责人
- 新负责人如果原本不在项目成员中，转交或调整负责人时必须自动加入项目成员
- 项目负责人可以主动转交负责人身份，但转交目标必须是项目经理角色用户
- 负责人转交后，原负责人保留项目成员身份
- 普通成员只能更新自己负责的子任务进度
- 管理员可以查看全部项目、阶段节点、子任务和用户；对团队项目可直接执行项目内操作，但仍需遵守统一业务规则限制
- 对普通员工创建的个人自用项目，管理员仅保留查看和删除权限，不允许执行项目内部修改动作
- 阶段节点顺序在项目内必须唯一
- 阶段节点顺序表示计划推进顺序，不表示系统级强依赖关系
- 已完成的阶段节点允许单独执行“撤销完成”
- 对某一阶段节点执行“撤销完成”时，不自动联动回退后续阶段节点状态
- 项目、阶段节点、子任务都必须维护时间区间，用于提供甘特图展示数据
- 项目完成和阶段完成都属于“手动确认动作”，不采用系统自动完结
- `已完成` 是最高优先级状态；一旦点击完成，对象状态立即进入 `已完成`
- 对尚未完成的对象，若按北京时间（`Asia/Shanghai`）判断当前日期已进入其 `planned_end_date` 的次日 `00:00` 及以后，即当前北京时间日期 `>` `planned_end_date`，则自动判定为 `已延期`
- 对尚未完成且未延期的对象，只有在满足开始信号后才进入 `进行中`；否则保持 `未开始`
- 到达项目、阶段节点、子任务的 `planned_start_date` 本身，不自动触发状态进入 `进行中`
- 所有基于 `DATE` 字段的状态判定统一按北京时间（`Asia/Shanghai`）计算，不使用 UTC 日期边界
- 子任务的开始信号包括：责任人手动点击开始、首次提交大于 `0` 的进度、项目负责人手动将其设为 `进行中`
- 阶段节点无子任务时，由项目负责人手动点击开始；有子任务时，只有当阶段当前状态为 `未开始`，且任一子任务进入 `进行中 / 已延期 / 已完成`，该阶段才自动进入 `进行中`
- 项目无阶段节点时，由项目负责人手动点击开始；有阶段节点时，只有当项目当前状态为 `未开始`，且任一阶段节点进入 `进行中 / 已延期 / 已完成`，该项目才自动进入 `进行中`
- 下级对象的自动联动只允许将父对象从 `未开始` 推进为 `进行中`，不会自动将父对象改写为其他状态
- 对已完成对象执行“撤销完成”后，不固定回到 `进行中`，而是按当前时间和开始信号重新计算为 `未开始 / 进行中 / 已延期`

## 9. 第一版功能范围

### 9.1 登录与访问

- 用户登录
- 基于身份进入各自可访问页面

### 9.2 项目列表页面

- 列出当前用户可见的全部项目
- 展示项目概要信息
- 新增项目
- 删除项目
- 按项目名称模糊查找项目
- 按项目状态筛选项目
- 展示项目甘特图

### 9.3 项目详情页面

- 按顺序展示项目阶段节点
- 点击阶段节点后展示节点详情卡片
- 新增阶段节点
- 删除阶段节点
- 通过拖拽调整阶段节点顺序
- 展示阶段节点甘特图
- 展示项目成员甘特图

### 9.4 项目成员管理

- 查看项目成员
- 添加项目成员
- 移除项目成员

### 9.5 子任务管理

- 在阶段节点下新增子任务
- 编辑子任务
- 删除子任务
- 为子任务指定负责人
- 重新分配子任务负责人
- 设置子任务 `planned_start_date` 和 `planned_end_date`
- 查看子任务状态、进度和备注

### 9.6 我的子任务

- 查看当前用户负责的全部子任务
- 按状态查看自己的子任务
- 进入单个子任务的进度页面

### 9.7 子任务进度页面

- 提交子任务进度更新
- 填写备注或进度说明
- 查看历史进度记录

### 9.8 模板能力

- 创建项目时支持选择系统预置模板
- 系统预置模板可一次性生成多个预设阶段节点
- 第一版仅提供系统预置模板，不支持用户自定义模板管理

### 9.9 管理员功能

- 管理员专属管理界面
- 查看用户详情
- 维护用户基础信息

## 10. 第一版明确不做的内容

- 自定义工作流引擎
- 子任务树形结构

- 阶段节点之间的复杂依赖关系
- 非线性节点拓扑结构

## 11. 非功能性要求

- 界面优先保证简洁和可用
- 第一版优先保证核心流程跑通，再逐步优化扩展性
- 系统响应速度满足企业内部日常使用即可
- 数据持久化存储在 PostgreSQL
- 部署环境为内部服务器
- 使用 Nginx 作为访问入口和反向代理
- 甘特图的主要可视化工作由前端完成，后端负责提供项目、阶段节点、子任务和成员维度所需的时间数据

## 12. 当前边界总结

本项目当前定位为“企业内部轻量项目进度跟踪工具”，核心在于：

- 用项目承载整体目标
- 用阶段节点承载阶段规划和阶段推进
- 用子任务承载具体执行工作
- 用进度记录承载推进历史
- 用甘特图承载时间视角下的可视化查看

第一版不追求复杂平台化能力，重点是把“创建项目、规划阶段、分配任务、推进进度、查看甘特图、完成项目”这条主流程做扎实。
