# 子任务模块临时并发策略

## 目的

这份文档先记录当前对子任务模块写侧并发控制的建议，作为后续实现时的临时准则。
当前目标不是把锁加满，而是和现有 `project`、`project_node`、`project_member` 的写侧规则保持一致，避免子任务模块单独长出另一套并发模型。

## 当前推荐策略

- 子任务模块继续采用锁。
- 但不采用“涉及多少表就锁多少表”的写法。
- 项目内所有写接口，统一先锁 `project` 行，把它作为同一项目下写操作的串行化根。
- 在已经锁住 `project` 的前提下，只额外锁当前真正要稳定的目标资源行。
- 依赖未锁定集合的条件，不要在 `Service` 中先查后判，统一压回最终原子 SQL。

一句话概括：

`project` 作为统一写根，固定锁顺序为“`project -> 当前目标行`”，集合条件尽量收敛到最后一条 `UPDATE / DELETE / INSERT ... SELECT / RETURNING`。

## 为什么继续用锁

现有实现已经基本是这个方向：

- `project_node` 写接口会先锁 `project`，再锁目标 `project_node`
- 节点完成条件不会在 `Service` 里先查子任务集合，而是压进最终原子 `UPDATE`
- 项目成员移除也不会先查“是否还有未完成子任务”，而是把条件压进最终 `DELETE`

如果子任务模块改成另一套并发策略，会直接影响这些已存在的父级规则：

- 项目完成后的冻结
- 节点完成后的冻结
- 节点时间区间必须覆盖全部子任务
- 项目成员移除时不能仍有未完成子任务
- 子任务写成功后推动父节点、项目进入进行中

因此当前阶段更重要的是“统一锁边界”，不是机械减少 `FOR UPDATE` 的数量。

## 锁边界总原则

### 1. 读接口不加锁

这些接口默认不需要事务锁：

- 获取节点下子任务列表
- 获取我的子任务列表
- 获取子任务详情
- 获取子任务进度记录列表

读侧继续优先保证实现简单，按现有项目风格直接查库即可。

### 2. 写接口统一走事务

这些接口统一走 `Controller -> Service -> Repository`，并在 `Service` 中开启事务：

- 创建子任务
- 修改子任务基础信息
- 删除子任务
- 手动开始子任务
- 撤销子任务完成
- 提交子任务进度

进入事务前，先完成以下边界校验：

- 登录态
- JSON 结构
- 字段类型
- 日期字符串格式
- 枚举值范围

不要把这类 HTTP 边界校验放到事务里占锁。

### 3. 固定锁顺序

当前建议统一使用以下顺序：

1. 先锁 `project`
2. 再锁当前目标行

“当前目标行”的含义按接口不同而不同：

- 创建子任务：目标行是 `project_node`
- 修改 / 删除 / 开始 / 撤销完成 / 提交进度：目标行是 `sub_task`

不建议在同一条链路里随手再扩锁到整批 `sub_task`、整批 `sub_task_progress` 或整批 `project_member`。

## 各类接口建议

### 创建子任务

建议顺序：

1. 锁 `project`
2. 锁目标 `project_node`
3. 读取并校验：
   - 项目是否存在
   - 节点是否存在
   - 项目是否已完成
   - 节点是否已完成
   - 操作者是否有权限
   - 子任务日期是否落在节点日期范围内
   - `responsible_user_id` 是否属于当前项目成员
4. 执行 `INSERT sub_task`

说明：

- 这里还没有 `sub_task` 行，因此不需要子任务级锁。
- `responsible_user_id` 是否属于项目成员，可以在已锁住 `project` 的前提下检查，不必额外锁 `project_member`。
- 如果后续决定“创建子任务后自动推动父节点/项目进入进行中”，这一步应写成幂等更新，而不是前置读取后再判断。

### 修改子任务基础信息

建议顺序：

1. 通过 `subtask_id` 先锁所属 `project`
2. 再锁目标 `sub_task`
3. 校验：
   - 项目是否存在
   - 子任务是否存在
   - 项目是否已完成
   - 所属节点是否已完成
   - 子任务自身是否已完成
   - 操作者是否有权限
4. 执行最终原子 `UPDATE`

这条最终 `UPDATE` 自己应继续保护以下条件：

- 日期区间本身合法
- 新日期仍被所属节点日期覆盖
- 新责任人仍属于当前项目成员

如果写返回空，再统一映射成冲突类错误，不要依赖事务前半段的旧读取抢先下结论。

### 删除子任务

建议顺序：

1. 锁 `project`
2. 锁目标 `sub_task`
3. 校验：
   - 项目是否存在
   - 子任务是否存在
   - 项目是否已完成
   - 所属节点是否已完成
   - 子任务是否已完成
   - 操作者是否有权限
4. 执行 `DELETE sub_task`

说明：

- `sub_task_progress` 通过外键级联删除，不需要单独加锁历史表。
- 删除动作本身是一条原子语句，尽量不要先查历史记录再删。

### 手动开始子任务

建议顺序：

1. 锁 `project`
2. 锁目标 `sub_task`
3. 校验：
   - 项目是否存在
   - 子任务是否存在
   - 项目是否已完成
   - 所属节点是否已完成
   - 子任务当前不是已完成
   - 操作者是子任务负责人，或具备项目负责人权限
4. 更新 `sub_task` 当前值
5. 插入一条 `sub_task_progress` 开始记录
6. 幂等推动父节点、项目进入进行中

说明：

- 当前值更新和历史记录写入必须放在同一个事务中。
- 只要成功写过当前值或历史值，后续若要整体放弃，就应显式 `rollback()`。
- 父级联动只允许把 `未开始 -> 进行中`，空返回不应视为失败。

### 撤销子任务完成

建议顺序：

1. 锁 `project`
2. 锁目标 `sub_task`
3. 校验：
   - 项目是否存在
   - 子任务是否存在
   - 项目是否已完成
   - 所属节点是否已完成
   - 子任务当前必须是已完成
   - 操作者是否有权限
4. 基于 `sub_task_progress` 读取：
   - 最近一条未完成状态历史
5. 更新 `sub_task` 当前值，恢复到正确的未完成状态

说明：

- 这里读取历史记录是安全的前提，是所有会修改当前值和追加历史的接口都先锁同一条 `sub_task`。
- 若不存在任何未完成历史，就回退到 `未开始 + 0%`；随后再按北京时间补一次是否延期的判断。
- 单个子任务的历史集合可以靠子任务行锁稳定住；这和“节点下全部子任务集合”是两类问题，不要混在一起。

### 提交子任务进度

建议顺序：

1. 锁 `project`
2. 锁目标 `sub_task`
3. 校验：
   - 项目是否存在
   - 子任务是否存在
   - 项目是否已完成
   - 所属节点是否已完成
   - 子任务当前不是已完成
   - 默认只有负责人可提交进度
   - 请求里的 `status` 和 `progress_percent` 匹配
4. 判断开始信号是否已经成立
5. 更新 `sub_task` 当前值
6. 插入一条 `sub_task_progress`
7. 幂等推动父节点、项目进入进行中

说明：

- “是否已存在开始信号”依赖的是当前这一个子任务的历史集合，可以在锁住该 `sub_task` 后安全读取。
- 一旦开始信号已经成立，不应允许通过提交进度把状态回退为 `未开始`。
- 当前值更新与历史追加必须同事务完成。

## 哪些条件不要提前在 Service 中判死

下面这类条件，如果依赖的是会被并发写改变的集合，不要在 `Service` 中先查后直接返回更具体的 `409`：

- 节点下是否还有未完成子任务
- 项目成员名下是否还有未完成子任务
- 节点时间收缩后是否仍覆盖全部子任务

这类条件应继续压在最终原子 SQL 中，由写语句自己决定能否成功。

原因是 PostgreSQL 默认是 `READ COMMITTED`：

- 同一事务里的两次 `SELECT` 不天然共享同一快照
- 如果没有一起锁住整组相关行，前一次读取结论可能在下一条写 SQL 之前已经过期

## 哪些资源不建议主动扩锁

当前不建议默认扩锁以下资源：

- 整个节点下所有 `sub_task`
- 某个项目下所有 `sub_task`
- 整个 `sub_task_progress` 历史集合对应的多行
- 整个项目的全部 `project_member`

原因：

- 锁范围过大，容易把同项目下原本可以快速完成的写操作全部拖慢
- 更容易形成死锁
- 很多业务条件本来就更适合放进最终原子 SQL，而不是靠先锁全量集合求稳定

## 父级联动策略

子任务写成功后，对父级的联动建议统一做成幂等更新：

- 任一子任务进入 `进行中 / 已延期 / 已完成`，可推动父节点从 `未开始 -> 进行中`
- 任一阶段节点进入 `进行中 / 已延期 / 已完成`，可推动项目从 `未开始 -> 进行中`

注意：

- 只做单向推进，不自动把父级推进到 `已完成`
- 不要求先查父级当前状态再决定是否更新
- 直接执行带条件的原子 `UPDATE ... WHERE status = 1`
- 空结果通常表示父级本来就不需要推进，不应作为失败处理

## 事务与回滚

子任务模块里，以下接口都属于“多段写但必须全有或全无”：

- 手动开始子任务
- 提交子任务进度
- 可能带父级联动的撤销完成

因此建议：

- 第一条真正写 SQL 之前，尽量完成全部读取和校验
- 第一条写 SQL 之后，如果后续任何一步失败且语义上要整段放弃，显式 `rollback()`
- 不要依赖作用域结束去猜测是否按业务意图回滚

## 当前实现阶段的建议

子任务模块实现时，优先按下面顺序推进：

1. 先补 repository 的带锁检查方法设计
2. 再定 `start / reopen / progress` 三类写接口的事务编排
3. 最后补读接口和 `.http` 用例

当前不建议一开始就做“减少锁数量”的优化。
先把并发模型统一，再在项目完成后整体 review 哪些锁边界可以收窄，会更稳。
